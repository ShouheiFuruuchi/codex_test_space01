# PDFバーコード抽出から商品マスタ紐づけ・原価集計までの仕組み開発ログ

## 目的

社販タグ表PDFからJANコードを抽出し、商品マスタCSVと紐づけて、社員別の原価合計を自動集計する仕組みを作る。

このMDは、後でnote記事化するための素材です。別のAIに読み込ませて記事化しやすいように、構成、実装方針、フォルダ構成、再現用プロンプト、注意点をまとめています。

## 作ったものの概要

ブラウザだけでPDF内バーコードを読む方式では精度が足りなかったため、Pythonバックエンドを追加した。

処理フローは以下。

1. PDFをアップロードする
2. バックエンドでPDFを高解像度画像に変換する
3. ZXingでEAN-13バーコードを検出する
4. 複数の画像前処理をかけて取りこぼしを減らす
5. JANコードを商品マスタCSVのB列と照合する
6. 商品マスタCSVのS列を原価として取得する
7. 社員別に数量、照合済み点数、未照合点数、原価合計を集計する

## なぜバックエンド化したか

最初はブラウザ標準の `BarcodeDetector` とPDF.jsで読取を試した。

しかし、以下の理由で精度が不足した。

- ブラウザのバーコード検出エンジンに依存する
- PDF内のバーコードがかすれている
- 手書き文字や罫線がバーコードに重なっている
- スキャンPDFの解像度やコントラストにばらつきがある
- 一部PDFはバーコード画像ではなく請求書形式だった

そこで、専用ライブラリ `zxing-cpp` を使うPythonバックエンドに切り替えた。

## バーコード抽出の考え方

PDFを画像化して、その画像に対して複数の前処理をかける。

実装した前処理:

- original
- autocontrast
- equalized
- sharpened
- binary_160
- binary_190

さらに、ZXingのbinarizerも複数試す。

- LocalAverage
- GlobalHistogram
- FixedThreshold
- BoolCast

バーコード形式はEAN-13に絞った。

理由:

- 今回のJANコードは13桁
- 形式を絞ると誤検出を減らせる
- 余計なQRやCode128などを探さなくて済む

## 重要だった改善ポイント

### 1. 画像前処理を複数試す

加藤さんPDFでは、最初は4件しか読めなかった。

画像を確認すると、実際には7件のバーコードがあった。

取りこぼしていた3件は、手書きや罫線、かすれが重なっていた。

そこで、`equalized`、`sharpened`、複数binarizerを追加したところ、7件すべて読めるようになった。

### 2. 重複カウントを防ぐ

同じバーコードを複数の前処理で読めることがある。

そのまま足すと数量が水増しされる。

そのため、以下の条件でクラスタリングした。

- 社員名
- JANコード
- PDFファイル名
- ページ番号
- バーコード位置

位置が近いものは同一バーコードとして扱う。

### 3. 商品マスタの列指定を明確にする

今回の指定:

- JANコード: B列
- 原価: S列

実装上は0始まりのインデックスなので、

```python
MASTER_BARCODE_COL = 1
MASTER_COST_COL = 18
```

とした。

商品名はC列を使った。

```python
MASTER_NAME_COL = 2
```

注意:

CSVヘッダー上は「原価」という列名がL列に見える場合がある。今回はユーザー指定を優先してS列を原価として扱った。

## API設計

### 商品マスタ取得

```text
GET /api/master
```

レスポンス例:

```json
{
  "ok": true,
  "count": 23204,
  "columns": {
    "barcode": "B",
    "name": "C",
    "cost": "S"
  },
  "items": [
    {
      "barcode": "2900000195455",
      "name": "商品名",
      "price": 1573,
      "cost": 1573
    }
  ]
}
```

### PDFバーコード解析

```text
POST /api/scan-pdfs
```

multipart/form-dataでPDFを送る。

レスポンス例:

```json
{
  "ok": true,
  "items": [
    {
      "employee": "加藤",
      "barcode": "2900000195455",
      "quantity": 1,
      "fileName": "(加藤)2026年4月社販表.pdf"
    }
  ],
  "files": [
    {
      "fileName": "(加藤)2026年4月社販表.pdf",
      "employee": "加藤",
      "count": 7
    }
  ]
}
```

### サンプルPDF自動集計

```text
GET /api/scan-samples
```

`社販表サンプル/` フォルダ内のPDFをバックエンド側で一括解析し、WEB画面へ反映するためのAPI。

当初はバックエンド単体では集計できていたが、WEB画面には自動反映されていなかった。理由は、画面側が「ユーザーがPDFを選択して読取ボタンを押す」前提だったため。

対策として、以下の流れに変更した。

1. ページ起動時に `GET /api/master` で商品マスタを自動読込
2. 商品マスタ読込後、PDF未選択なら `GET /api/scan-samples` を実行
3. 返却されたJAN一覧を `barcode-input` 相当の内部データに反映
4. そのまま商品マスタ照合と社員別集計を実行

これにより、`http://127.0.0.1:8000` を開くだけでサンプルPDFの集計結果が画面に表示されるようになった。

注意:

```text
file:// で index.html を直接開くとAPIが動かない。
必ず python backend.py を起動し、http://127.0.0.1:8000 で開く。
```

## 未対応・注意点

### 請求書形式PDFはOCRが必要

渡辺さんPDFはバーコードタグ表ではなく、請求書形式だった。

この場合、バーコード画像が存在しないため、バーコード認識では抽出できない。

必要な対応:

- OCRを導入する
- 明細表の左側に印字された商品コードを読む
- JANではなく商品コード照合に対応する

候補:

- Tesseract OCR
- Google Cloud Vision
- Azure AI Vision
- OpenAI Vision API

### 処理速度と精度のトレードオフ

最初の高速設定では全体約12秒だったが、加藤さんPDFの3件を取りこぼした。

精度改善後は全体約26秒になった。

WEB起動時にサンプルPDF自動集計を走らせる場合、初期表示から集計反映まで数秒から十数秒かかることがある。画面が0件に見えても、バックエンド処理中の可能性があるため、ステータス表示やトーストで処理中であることを伝えるとよい。

実務では以下のモード分けがよい。

- 高速モード: 日常処理用
- 高精度モード: 取りこぼし確認用
- OCRモード: 請求書形式PDF用

### 傾いたバーコードの補正

3月分の工藤さんPDFで、傾いたバーコードが `6940008196012` と誤読され、商品マスタ未照合になった。

画像を確認すると、右端のタグが斜めに貼られており、バーコードの一部が歪んでいた。そこで、通常読取で以下のような「怪しいJAN」が出た場合だけ、傾き補正リトライを行うようにした。

怪しいJANの判定例:

```text
13桁だが商品マスタに存在しない
かつ、社販タグで多い 290... 系ではない
```

補正処理:

```text
-10度、-6度、-2度、+2度、+6度、+10度で小角度回転
autocontrast + sharpened を試す
EAN-13専用で再読取
FixedThreshold binarizerで読む
商品マスタに存在するJANだけ採用
既存JANと重複するものは追加しない
```

この処理は全ページに毎回かけると重くなるため、通常読取で未照合・怪しいJANが出たページだけ実行する設計にした。

結果:

```

```

## 重要な学び

- PDFバーコード読取は、ただライブラリを使うだけでは取りこぼす
- スキャン品質、手書き、罫線、かすれが精度に大きく影響する
- 画像前処理と複数binarizerの組み合わせが効く
- ただし、複数処理で読んだ結果をそのまま足すと数量が増える
- 重複排除は必須
- 商品マスタの列指定は明確にする
- CSVはShift-JIS/CP932の可能性が高い
- バーコード画像がないPDFはOCRが必要
- 高速モードと高精度モードを分けると実務で使いやすい
- バックエンドで集計できていても、WEB画面に反映するAPIと起動時フローがなければユーザーには0件に見える
- ローカルWEBアプリは `file://` ではなく、必ずローカルサーバーURLで確認する
- 傾いたバーコードは未照合や誤読の原因になる
- 傾き補正は有効だが重いので、未照合・怪しいJANが出た場合だけ走らせるとバランスがよい
- 補正結果は商品マスタ照合を通してから採用すると、誤検出を増やしにくい
